# UNCITRAL ISDS 谈判文本变量设计说明

## 一、这份文档的用途

这份文档说明：在整理 UNCITRAL ISDS 改革谈判发言材料时，**哪些变量必须保留、这些变量分别是什么意思、以及为什么它们对后续研究不可或缺**。

一句话概括：

> 没有一套清楚、稳定、可复核的变量体系，后面的 LLM 标注、人工校验、联盟网络构建和实证解释都会不稳。

---

## 二、变量设计的基本原则

整理发言数据时，建议遵循三个原则：

1. **先保证可追踪**：每条记录都必须能回到原始文本。
2. **再保证可比较**：不同会议、不同国家、不同议题之间要能放在同一个表里比较。
3. **最后保证可分析**：变量不仅要描述文本，还要支持后续的网络分析和解释性分析。

因此，变量不能只满足“把话记下来”，还必须满足“以后还能查、还能比、还能解释”。

---

## 三、建议保留的核心变量

下面这些变量建议作为基础数据表的核心字段。

| 变量名 | 是什么 | 为什么必须保留 |
|---|---|---|
| `record_id` | 每条发言记录的唯一编号 | 方便追踪、修改、复核，也方便把人工标注和 LLM 标注准确对齐。 |
| `source_doc` / `session_id` | 这条发言来自哪一场会议、哪份 transcript 或哪份会议材料 | 同一个国家会在多场会议中发言。没有这个变量，就无法进行跨会议比较，也无法分析时间变化。 |
| `date` | 发言发生的日期 | 这是动态分析的基础。后续如果要研究联盟演变、议题扩散或前后变化，这个变量必须存在。 |
| `text` | 原始发言文本 | 这是所有标注的证据基础。没有原文，就无法复核分类结果，也无法检查 LLM 是否误判。 |
| `speaker` | 谁在发言，通常是国家、区域组织或观察员 | 这是网络分析中的节点标识。没有 `speaker`，就无法构建国家间关系网络。 |
| `speaker_development` | 发言主体的发展水平类别，如发达国家、发展中国家、最不发达国家 | 用于解释联盟是否沿发展水平分化。它帮助研究者从“谁与谁站在一起”进一步走向“为什么站在一起”。 |
| `speaker_regime` | 发言国的政体类型，如民主程度、威权程度，可后续接入 V-Dem 等数据库 | 用于检验制度特征是否影响谈判联盟形成。 |
| `speaker_region` | 发言国所在地区，如亚洲、非洲、欧洲、拉丁美洲等 | 地区因素往往是谈判协调的重要来源，也是解释联盟结构的关键变量。 |
| `speaker_type` | 发言主体类型，如国家、主席、秘书处、观察员、行业组织 | 主席和秘书处通常不应直接纳入国家联盟网络；观察员也应与国家区分，否则会污染分析结果。 |
| `topic` | 这段发言讨论的是哪个议题 | 研究重点不是“所有话混在一起”，而是看不同议题上的立场结盟。没有 `topic`，就无法做分议题分析。 |
| `stance` | 发言者在该议题上的立场 | 这是判断两个国家是否立场接近、是否可能形成联盟的核心变量。 |
| `target_proposal` | 发言针对的是哪个具体提案、文本选项或制度设计 | 同一议题下常常有多个争点。没有这个变量，容易把不同对象上的立场误判为同一立场。 |
| `evidence_quote` | 支撑立场判断的原文引句或关键短段 | 这是控制 LLM 幻觉和保证研究透明度的关键证据字段。 |
| `topic_article` | 发言对应的是哪个条文、段落、工作文件部分或正式文本位置 | `topic` 是研究者归纳的议题标签，`topic_article` 是会议文本中的正式定位。两者并用，更方便回到原始制度结构。 |

---

## 四、这些变量分别解决什么研究问题

为了方便理解，可以把这些变量分成四组。

### 4.1 追踪与复核类变量

包括：`record_id`、`source_doc` / `session_id`、`date`、`text`、`evidence_quote`

这组变量的作用是：

- 确保每条记录都能回到原始文本；
- 确保后续修改和复核有据可查；
- 确保人工标注与 LLM 输出可以一一对应；
- 确保研究过程具有可复制性。

如果缺少这组变量，后面的标注再精致，也难以证明其可靠性。

### 4.2 行为主体类变量

包括：`speaker`、`speaker_type`、`speaker_development`、`speaker_regime`、`speaker_region`

这组变量的作用是：

- 确定“谁在说话”；
- 区分哪些主体应进入国家联盟网络；
- 为解释联盟形成机制提供结构性背景。

换句话说，这组变量让研究不止停留在“描述谁和谁相似”，而能进一步分析“哪些国家为什么更容易站在一起”。

### 4.3 议题与立场类变量

包括：`topic`、`stance`、`target_proposal`、`topic_article`

这组变量直接决定你的研究能否识别“隐形联盟”。

- `topic` 告诉我们大家在讨论什么；
- `target_proposal` 告诉我们大家针对的是哪一个具体争点；
- `stance` 告诉我们他们在该争点上的方向是什么；
- `topic_article` 帮助我们把分析重新放回正式文本结构。

如果没有这组变量，就很难区分：

- 是在同一议题上形成联盟；
- 还是只是碰巧都在同一场会议中发言；
- 是对同一个提案表态；
- 还是其实分别在谈不同制度细节。

## 五、建议的数据表最小结构

如果你要先搭一个“最小可用版本”的数据库，建议至少包含以下字段：

| 字段 | 是否必需 | 说明 |
|---|---|---|
| `record_id` | 必需 | 唯一主键 |
| `session_id` | 必需 | 会议或材料来源 |
| `date` | 必需 | 发言日期 |
| `speaker` | 必需 | 发言主体 |
| `speaker_type` | 必需 | 国家、主席、秘书处、观察员等 |
| `text` | 必需 | 原始发言文本 |
| `topic` | 必需 | 所涉议题 |
| `stance` | 必需 | 立场标签 |
| `target_proposal` | 强烈建议 | 具体针对的提案或文本点 |
| `evidence_quote` | 强烈建议 | 支撑判断的引文 |
| `topic_article` | 建议 | 对应条文或段落 |
| `speaker_region` | 建议 | 用于解释联盟结构 |
| `speaker_development` | 建议 | 用于检验发展水平分化 |
| `speaker_regime` | 建议 | 用于连接外部制度数据 |

如果研究时间有限，可以先完成“必需 + 强烈建议”部分，再逐步补充背景解释变量。

## 六、实务上的几个提醒

### 6.1 不要只保存摘要，要保存原文

即使后续会让 LLM 生成摘要，也不能替代原始 `text`。研究中的核心判断必须能回到原句。

### 6.2 `topic` 和 `target_proposal` 不能混用

- `topic` 是较高层级的议题类别；
- `target_proposal` 是更具体、可争论、可采纳的制度主张。

例如：

- `topic = code_of_conduct`
- `target_proposal = extend_confidentiality_obligation_after_service`

两者层级不同，不能合并成一个字段。

### 6.3 `speaker_type` 一定要单独保留

这不是“可有可无”的字段。因为 Chair、Secretariat、Observer 与国家代表的发言逻辑完全不同，若不区分，会直接影响网络节点定义。

### 6.4 `evidence_quote` 是控制质量的关键

如果只保留最终标签，不保留证据句，后面很难检查：

- 是不是 LLM 误解了语气；
- 是不是人工标注标准不一致；
- 是不是“支持”和“担忧”被混淆了。

---

## 七、一句话总结

如果把这套研究比作盖房子，那么变量设计就是地基。

> `record_id`、`source`、`text` 解决“能不能查回来”；  
> `speaker` 系列变量解决“是谁在表态”；  
> `topic`、`stance`、`target_proposal` 解决“他们在什么问题上是否站在一起”。

这三层一起完整，后面的 LLM 标注、联盟识别和实证解释才站得住。
